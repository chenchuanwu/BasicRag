# 食品安全投诉举报 RAG SaaS 可执行计划（V1）

## 1. 项目目标

8 周内把当前 RAG Demo 升级为可试运行的业务系统：

1. 输入投诉举报材料后，自动检索法规、案例、模板。
2. 生成可编辑、可追溯、有法条依据的回复文书草稿。
3. 所有结论默认需人工审核后才能出文。
4. 支持基础 SaaS 能力（租户隔离、审计日志）。

### 实施原则

- 先做知识底座，再做 Agent。
- 先解决可用性，再做复杂智能。
- 每阶段必须有可量化验收指标。

---

## 2. 当前基线（对应现有代码）

| 文件 | 已有能力 | 主要问题 |
|------|---------|---------|
| `knowledge_base.py` | 基础入库 + 文本切分 | 去重逻辑有重复写入 bug；metadata 字段不完整 |
| `vector_stores.py` | Chroma 向量检索 | 仅 Top-K，缺少 BM25/重排/metadata 过滤 |
| `rag.py` | 基础问答链路 | 缺少结构化文书输出 + 引用强约束 |
| `app_file_uploader.py` | Streamlit 上传页 | 声明支持 pdf/docx 但解析链路不可用 |
| `app_qa.py` | 聊天问答页 | 流式缓存未落盘导致历史消息丢失 |
| `file__history_store.py` | 文件会话历史 | 缺少租户隔离与审计字段 |
| `config_data.py` | 基础配置 | session_id 硬编码，缺少检索/租户参数 |

## 3. Phase 1（第 1-2 周）：知识底座与检索可靠性

### 交付目标

实现稳定入库、可控检索、可评测基线。

### 任务清单

**P1-1 文档解析统一化**

- 新增 `RAG/parsers/` 目录，统一接口 `parse(file) -> (text, metadata)`。
- `txt/md`：UTF-8 直接读取。
- `pdf`：`PyMuPDF`（扫描件预留 OCR 接口）。
- `docx`：`python-docx`。
- 改造 `app_file_uploader.py` 调用新解析器。

**P1-2 入库去重修复**

- 改造 `knowledge_base.py`：只保留一次入库路径（删除当前的双重 add_texts）。
- 使用 `doc_md5 + chunk_md5` 双层去重。
- 入库成功后才保存 md5 记录。

**P1-3 metadata 标准化**

- 每个 chunk 必须写入以下字段：

```json
{
  "tenant_id": "county_a",
  "doc_type": "law | case | template | sop",
  "source_name": "食品安全法",
  "article": "第三十四条",
  "violation_type": "标签 | 变质 | 异物 | 虚假宣传",
  "effective_date": "2025-01-01",
  "file_name": "xxx.pdf",
  "doc_md5": "xxx",
  "chunk_md5": "xxx"
}
```

**P1-4 Hybrid Retrieval V1**

- 改造 `vector_stores.py`：
  - 向量检索（dense）。
  - 关键词检索（BM25，可用 `rank_bm25` 或 LangChain BM25Retriever）。
  - 融合排序（RRF 或加权）。
- 增加基础 metadata 过滤（`doc_type`、`tenant_id`）。

**P1-5 建立评测基线**

- 在 `RAG/docs/` 构建 50 条标准问答集（含目标法条标注）。
- 指标：`Recall@5`、`Citation Hit Rate`。
- 编写可复跑评测脚本。

### 验收标准

- [ ] 上传 `txt/pdf/docx/md` 均可解析并入库
- [ ] 同文件重复上传不新增向量
- [ ] 法规类问题 Top-5 至少命中 1 条相关法条
- [ ] 输出评测报告（含基线分数）

## 4. Phase 2（第 3-4 周）：投诉理解与法规优先检索

### 交付目标

系统能理解投诉类型，并优先返回可用于执法回复的法规依据。

### 任务清单

**P2-1 投诉分类器**

- 新增 `RAG/classifier.py`。
- 分类维度：食品类型、问题类型、严重等级。
- 策略：规则优先 + LLM 补充；无法判断时返回 `unknown`。

**P2-2 多通道并行检索**

- 并行检索三个通道：
  - 法规库（`doc_type=law`）
  - 历史投诉回复（`doc_type=case`）
  - 文书模板（`doc_type=template`）
- 每个通道独立 top-k 与权重。

**P2-3 重排器**

- 新增 `RAG/reranker.py`。
- 排序目标：`law > regulation > case > template`。
- 先用轻量 cross-encoder 或 LLM 打分，后续可替换。

### 验收标准

- [ ] 分类器主类准确率 >= 80%（50 条样本）
- [ ] Top-3 结果中法规文档占比明显提升
- [ ] 检索日志可追踪每通道召回量与最终排序分

---

## 5. Phase 3（第 5-6 周）：结构化文书生成与合规控制

### 交付目标

从"问答"升级为"可直接编辑的投诉回复文书草稿生成"。

### 任务清单

**P3-1 固定结构生成**

- 改造 `rag.py` 提示词，强制 5 段输出：
  1. 投诉举报基本情况
  2. 调查核实情况
  3. 处理依据（必须含法条引用）
  4. 处理结论
  5. 告知事项
- 缺失依据时明确输出"依据不足，需人工补充"。

**P3-2 引用校验（Citation Guard）**

- 生成前注入"可引用法条池"（来自检索结果）。
- 生成后校验：
  - 每条法律结论是否有引用。
  - 引用条款是否在检索上下文中。
- 不满足则标记警告。

**P3-3 历史案例模式复用**

- 注入 top-k 相似案例的"语言风格 + 论证框架"。
- 禁止直接复制历史案件事实字段（防串案）。

**P3-4 审核流页面改造**

- 改造 `app_qa.py`：
  - 左侧：草稿正文（可编辑）。
  - 右侧：引用证据列表。
  - 底部：状态流转（草稿 / 已审 / 退回）。
- 修复流式回答 `ai_res_list` 未填充的 bug。

### 验收标准

- [ ] 文书结构完整率 >= 90%
- [ ] 法律结论引用覆盖率 >= 95%
- [ ] 错误引用率 <= 5%
- [ ] 审核人员可在页面编辑并切换状态

---

## 6. Phase 4（第 7-8 周）：Agentic 增强与 SaaS 基础治理

### 交付目标

构建可回溯的多步推理流程，并完成最小可用的 SaaS 治理能力。

### 任务清单

**P4-1 Agent 工作流编排**

- 固化 6 步流程：
  1. 投诉分类
  2. 检索策略决策
  3. 多通道检索
  4. 重排
  5. 文书生成
  6. 引用校验
- 每步写可检索日志，单步失败可重试。

**P4-2 Corrective RAG**

- 引用校验失败 / 证据不足时自动触发二次检索。
- 二次仍失败则降级为"需人工处理建议单"。

**P4-3 多租户与审计**

- 所有检索、会话、生成记录带 `tenant_id`。
- 审计字段：`operator`、`source_ids`、`approval_status`。
- 检索入口默认注入 `tenant_id` 过滤，禁止跨租户召回。

### 验收标准

- [ ] 任意输出可回放完整证据链
- [ ] 不同租户数据不串读
- [ ] 高风险场景错误强结论显著下降

## 8. KPI（上线前必须达标）

### 检索质量

- `Recall@5 >= 0.75`（法规问答集）
- `Citation Hit Rate >= 0.85`
- Top-3 法规占比 `>= 60%`

### 生成质量

- 文书结构完整率 `>= 90%`
- 法律结论引用率 `>= 95%`
- 错误引用率 `<= 5%`

### 业务效率

- 草稿一次通过率 `>= 70%`
- 单条草稿生成耗时 `<= 12s`（不含人工编辑）

---

## 9. 风险与控制

| 风险 | 控制措施 |
|------|---------|
| 历史案例质量不齐 | 入库前质量分层，仅高质量样本参与风格复用 |
| 法规更新滞后 | 法规版本字段 + 定时更新 + 过期降权 |
| 模型越权下结论 | 模板约束 + 引用校验 + 人工审核三道闸 |
| 多租户数据串读 | 检索入口统一注入 `tenant_id` 过滤 |
